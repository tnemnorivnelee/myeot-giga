# ADR-003: 백그라운드 감시 — WorkManager 선택

- **Status**: Accepted
- **Date**: 2026-06-30

## Context

PRD 요구사항: 앱 종료 상태에서도 주기적으로 데이터 사용량을 체크하고, 임계값 도달 시 알림 발송.

초기 요구사항은 "약 15분 주기"였으나, 정확한 주기 보장은 필수 요건이 아님을 확인.

후보: **WorkManager** vs **AlarmManager**

## 의사결정 과정

### 1단계: 타이밍 정확도 기준 검토

| | WorkManager | AlarmManager |
|--|--|--|
| 타이밍 정확도 | OS가 유연하게 조정 | 정확한 타이밍 가능 |
| 최소 주기 | 15분 | 제한 없음 |

"15분을 정확히 지킬 필요 없다"는 요구사항 확인 → 타이밍 정확도는 결정 기준에서 제외.

### 2단계: 타이밍과 무관한 AlarmManager 문제 검토

타이밍 정확도 외에도 AlarmManager를 피해야 하는 이유:

1. **재부팅 시 알람 소멸** — 기기 재부팅 후 등록한 알람이 모두 사라짐. `BOOT_COMPLETED` 브로드캐스트 수신 후 알람을 직접 재등록하는 코드 필요.
2. **앱 업데이트 후 알람 소멸** — 업데이트 시에도 동일하게 재등록 처리 필요.
3. **Doze 모드 대응** — `setAndAllowWhileIdle()`도 Doze 중 지연 가능. 제대로 하려면 `setExactAndAllowWhileIdle()` + 사용자에게 배터리 최적화 예외 권한 요청까지 필요.
4. **API 레벨별 동작 차이** — Android 버전마다 알람 동작이 달라 버전 분기 코드가 복잡해짐.

WorkManager는 위 네 가지를 모두 내부적으로 처리. 직접 구현 불필요.

## Decision

**WorkManager**를 사용한다. 주기는 배터리 영향 최소화를 위해 **30분**으로 설정.

## Consequences

**Good**
- 재부팅, 앱 업데이트, Doze 모드 자동 처리 — 별도 코드 불필요
- Google 공식 권장 방식으로 장기적 호환성 보장
- 구현 단순, 유지보수 용이

**Bad**
- OS가 주기를 지연할 수 있음 (요구사항상 허용 범위)
- 최소 주기 15분 제약 — 더 짧은 주기가 필요해질 경우 재검토 필요
